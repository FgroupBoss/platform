---

## 一、调研背景与目的

当前市场上，产研测协作的典型痛点是**工具割裂**——产品需求在文档里，研发任务在项目管理工具里，缺陷在测试表格里，版本发布靠会议同步。信息孤岛导致：
- 项目进度不透明
- 需求变更无法追溯影响面
- 缺陷与需求、代码关联断裂
- 多系统切换消耗大量沟通成本

**本次调研目标**：梳理市面主流产品能力，为自建平台提供功能参照和差异化定位依据。

## 三、核心竞品深度分析

### 3.1 Jira（Atlassian）

**市场地位**：全球市场占有率最高，敏捷开发事实标准。

**核心能力**：
- Epic → Story → Task → Sub-task 四级事务层级
- Scrum + Kanban 双模式原生支持
- 强大的 JQL（Jira Query Language）查询
- 数千个 Marketplace 插件扩展
- 与 GitHub/GitLab/Bitbucket 深度集成

**对标价值**：工作流自定义能力、权限模型、JQL 查询是核心参照。

**可借鉴点**：灵活的状态机、丰富的报表体系、Webhook 事件机制。

---

### 3.2 PingCode

**市场地位**：国内头部研发管理平台，小红书、长城汽车、华夏基金等客户。

**核心能力**：
- 需求管理：从客户反馈到需求池、评审、规划全流程
- 研发管理：Scrum / Kanban / 混合模式
- 测试管理：用例库、测试计划、执行记录、缺陷关联
- 全链路追溯：需求 ↔ 任务 ↔ 代码提交 ↔ 测试用例 ↔ 缺陷
- 效能度量：交付周期、吞吐量、缺陷趋势等

**对标价值**：全链路关联和追溯能力的实现方式，是本次自建平台的核心参照。

**可借鉴点**：对象之间的关联设计、效能度量指标体系、测试与研发一体化视角。

---

### 3.3 TAPD（腾讯）

**市场地位**：源于腾讯内部大规模敏捷实践，面向国内开发团队。

**核心能力**：
- 需求管理：需求池、需求评审、需求追踪
- 迭代管理：迭代规划、进度跟踪、燃尽图
- 缺陷管理：缺陷报告、缺陷跟踪、质量分析
- 测试管理：测试计划、测试用例、测试执行
- 与企微、腾讯会议等生态深度集成

**对标价值**：开箱即用的体验设计、敏捷迭代管理流程。

**可借鉴点**：迭代燃尽图、需求评审流程、看板视图。

---

### 3.4 禅道（ZenTao）

**市场地位**：国产开源项目管理标杆，用户基数最大。

**核心能力**：
- 产品 → 项目 → 执行 → QA → 发布 完整闭环
- Scrum + 瀑布双模式
- 内置测试管理（Bug / 用例 / 版本关联）
- 开源版免费、可自行部署

**对标价值**：全流程闭环思路、测试管理内置设计。

**可借鉴点**：需求与测试用例的关联方式、Bug 管理流程。

---

### 3.5 华为云 CodeArts

**核心能力**：
- 需求管理（Req）：IPD/DevOps/看板多模型，需求全链路追溯
- 测试管理（TestPlan）：启发式测试设计、亿级用例管理、自动化加速
- 效能洞察（Board）：四大角色视角、200+ 指标库

**对标价值**：需求-设计-代码-用例端到端追溯、效能度量体系。

**可借鉴点**：需求四层分解（需求-场景-测试点-用例）、门禁机制。

## 四、行业最佳实践总结

### 4.1 全链路追溯是核心差异化能力

一线平台（PingCode、CodeArts、禅道）的共同特征：
- 需求 ← 关联 → 开发任务
- 开发任务 ← 关联 → 代码提交
- 需求 ← 关联 → 测试用例
- 测试用例 ← 关联 → 缺陷
- 缺陷 ← 关联 → 修复任务
- 需求 ← 关联 → 发布版本

### 4.2 效能度量体系趋于成熟

| 度量维度 | 关键指标 |
|----------|----------|
| 交付效率 | 需求交付周期、迭代准时率、吞吐量 |
| 交付质量 | 缺陷逃逸率、测试通过率、线上事故数 |
| 交付能力 | 需求变更率、自动化覆盖率、代码复用率 |
| 交付价值 | 需求 ROI、客户满意度 |

### 4.3 工作流自定义是必备能力

几乎所有成熟平台都支持：
- 自定义状态机（需求/任务/缺陷独立流转）
- 自定义字段
- 自定义审批节点
- 自动化规则引擎

### 4.4 通知与集成是效率倍增器

- 站内信 + 企微/飞书/钉钉通知
- Git 代码仓集成（自动关联 Commit）
- CI/CD 集成（构建/部署状态同步）
- Open API + Webhook

## 五、我们的差异化定位

| 维度 | 现有产品普遍痛点 | 我们的定位 |
|------|------------------|-----------|
| 追溯矩阵 | 多为单向关联，缺少全局可视图 | **双向追溯矩阵视图**，以需求/缺陷任意视角快速追溯 |
| 角色体验 | 多为通用界面，各角色入口混杂 | **角色专属工作台**，产品/研发/测试/管理者各有专属视图 |
| 轻量度 | Jira 太重，Linear 太轻（无测试） | **适中复杂度**，覆盖产研测全链路但不臃肿 |
| 度量 | 指标分散，缺少"一句话"结论 | **项目健康度评分**，让管理者一眼看懂项目状态 |
| 部署 | 国产平台多偏向私有化或纯 SaaS | **灵活部署**，支持 SaaS + 私有化 |

## 六、结论与建议

1. **核心参照**：PingCode 的全链路追溯设计 + TAPD 的易用性 + CodeArts 的效能度量体系
2. **关键差异化**：角色专属工作台 + 双向追溯矩阵 + 项目健康度评分
3. **MVP 范围**：需求管理 + 任务管理 + 缺陷管理 + 迭代管理 + 追溯矩阵
4. **二期扩展**：测试管理 + 效能度量 + 自动化规则 + 知识库
