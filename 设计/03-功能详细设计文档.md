## 二、核心状态机设计

### 2.1 需求状态流转

```
          ┌──────────────────────────────────────┐
          │                                      │
          ▼                                      │
  ┌──────────┐  评审通过   ┌──────────┐  开发开始   ┌──────────┐
  │  待评审   │ ─────────→ │  已评审   │ ─────────→ │  开发中   │
  └──────────┘            └──────────┘            └─────┬────┘
       │                       │                       │
       │ 评审驳回               │ 打回                   │ 提测
       ▼                       ▼                       ▼
  ┌──────────┐            ┌──────────┐            ┌──────────┐
  │  已关闭   │            │  待评审   │            │  测试中   │
  └──────────┘            └──────────┘            └─────┬────┘
       ▲                                                │
       │                                                │ 测试通过/不通过
       │                                                ▼
       │                      ┌──────────┐  发布    ┌──────────┐
       └──────────────────────│  已发布   │ ←────── │  已关闭   │
            关闭(任意阶段)     └──────────┘         └──────────┘
```

**状态变更规则**：

| 当前状态 | 允许变更到 | 触发条件 | 自动操作 |
|----------|-----------|----------|----------|
| 待评审 | 已评审 | 评审通过 | 通知研发负责人 |
| 待评审 | 已关闭 | 评审驳回/取消 | - |
| 已评审 | 开发中 | 关联任务开始开发 | 更新迭代进度 |
| 已评审 | 待评审 | 打回重审 | 通知产品经理 |
| 开发中 | 测试中 | 所有关联任务提测 | 通知测试负责人 |
| 测试中 | 已发布 | 测试通过且发布 | 通知产品经理 |
| 测试中 | 开发中 | 测试不通过 | 通知研发负责人 |
| 任意 | 已关闭 | 手动关闭 | 清理关联迭代 |

### 2.2 研发任务状态流转

```
  ┌──────────┐   开始开发   ┌──────────┐   开发完成   ┌──────────┐
  │  待开始   │ ──────────→ │  开发中   │ ──────────→ │  已完成   │
  └──────────┘             └──────────┘             └─────┬────┘
                                                          │
                                                          │ 提测
                                                          ▼
                                                     ┌──────────┐
                                                     │  已提测   │
                                                     └──────────┘
```

**状态变更规则**：

| 当前状态 | 允许变更到 | 触发条件 |
|----------|-----------|----------|
| 待开始 | 开发中 | 开发者认领并开始 |
| 开发中 | 已完成 | 开发完成 |
| 已完成 | 已提测 | 提交测试 |
| 已完成 | 开发中 | 测试发现缺陷，打回 |
| 任意 | 待开始 | 重置 |

### 2.3 缺陷状态流转

```
  ┌──────────┐   确认    ┌──────────┐   开始修复   ┌──────────┐
  │   新建    │ ────────→ │   确认    │ ──────────→ │  修复中   │
  └────┬─────┘           └──────────┘             └─────┬─────┘
       │                                                │
       │ 拒绝(关闭)                                      │ 修复完成
       ▼                                                ▼
  ┌──────────┐         ┌──────────┐   验证通过   ┌──────────┐
  │   关闭    │ ←────── │  已验证   │ ←────────── │  已修复   │
  └──────────┘         └─────┬────┘             └──────────┘
       ▲                      │
       │                      │ 验证不通过
       │                      ▼
       │                 ┌──────────┐
       └─────────────────│  修复中   │
           重新打开       └──────────┘
```

## 五、通知系统设计

### 5.1 通知触发规则

| 触发事件 | 接收人 | 通知内容模板 |
|----------|--------|-------------|
| 需求状态变更为"已评审" | 研发负责人 | 「{需求标题}」已评审通过，请开始开发 |
| 需求状态变更为"测试中" | 测试负责人 | 「{需求标题}」已提测，请安排测试 |
| 需求被打回 | 产品经理 | 「{需求标题}」被 {操作人} 打回 |
| 开发任务分配 | 被分配人 | {操作人} 将任务「{任务标题}」分配给你 |
| 缺陷被创建 | 研发负责人 | 新缺陷「{缺陷标题}」({严重程度})，请确认 |
| 缺陷被指派 | 被指派修复人 | 缺陷「{缺陷标题}」指派给你修复 |
| 缺陷修复完成 | 创建人/验证人 | 缺陷「{缺陷标题}」已修复，请验证 |
| 评论 @提及 | 被 @ 的人 | {操作人} 在「{对象}」中 @了你 |

## 六、操作日志设计

所有关键操作均需记录日志，格式如下：

| 字段 | 说明 |
|------|------|
| 操作人 | 执行操作的用户 |
| 操作类型 | 创建/更新/删除/状态变更/分配 |
| 操作对象 | 需求/任务/缺陷/用例 |
| 对象 ID | 操作的实体 ID |
| 变更内容 | JSON 格式，记录变更前后对比 |
| 操作时间 | 精确到秒 |

**变更内容示例**：
```json
{
  "field": "status",
  "from": "待评审",
  "to": "已评审"
}
```
